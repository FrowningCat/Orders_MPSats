Как считаются графики

Логика и источники данных для каждого блока дашборда

Заказы по неделям — год к году

Источник — таблица wb_orders_dashboard, заполняется через CSV-отчёт воронки продаж WB (эндпоинт nm-report/downloads). Заказы группируются по ISO-неделям и сравниваются 2025 год к 2026-му на одной шкале недель.

  • Категория берётся из WB_Product_Cards.subjectName по nm_id
  • Неделя считается по ISO-стандарту прямо из даты заказа — не по полю week_num в таблице, чтобы избежать рассинхрона на границе декабря/января
  • Точка на графике = сумма заказов (шт) за всю неделю по выбранным категориям

Только наши артикулы — не фильтрует график, а добавляет две дополнительные линии зелёным цветом (сплошная — 2025, пунктир — 2026), показывающие только заказы по вашим SKU, поверх общего объёма.

  • «Наш» артикул определяется через цепочку: wb_orders_dashboard.nm_id → WB_Product_Cards.vendorCode → сверка с WB_Working_Articles.article
  • Джойн напрямую по vendor_code из wb_orders_dashboard не используется — это поле там пустое

Динамика суммарной частоты запросов MPStats - год к году

Источник — таблица mpstats_kw_dashboard (частотность поисковых запросов WB по неделям), агрегированная по категории (item_name). Сравнение идёт так же, как и с заказами — по ISO-неделям, 2025 к 2026.

  • Без активного поиска — суммарная частота всех запросов категории (total_freq из kw_weekly)
  • При активном поиске «Поиск по запросу» — график переключается на данные из «Топ-50» (конкретные совпавшие запросы), а не всю категорию, чтобы цифры на графике совпадали с тем, что видно в таблице снизу

Динамика частоты запросов MPStats

Тот же источник (mpstats_kw_dashboard), но с разбивкой по типу частотности:

  • ВЧ — высокочастотные запросы
  • СЧ — среднечастотные
  • НЧ — низкочастотные

При активном поиске по запросу график по-прежнему показывает разбивку ВЧ/СЧ/НЧ — теперь частоты для каждого типа считаются по конкретным совпавшим запросам (не по всей категории), поэтому цифры точно соответствуют тому, что видно в «Топ-50» ниже.

Заказы vs частота запросов

Совмещает на одном графике факт продаж (orders_daily) и спрос по MPStats (kw_weekly/top_keywords) — с двумя независимыми шкалами Y слева и справа, потому что абсолютные величины несравнимы (заказы — десятки/сотни штук, частота запросов — тысячи/миллионы).

  • Единая хронологическая лента (как «Динамика частоты запросов MPStats»), а не сравнение 2025 vs 2026 бок о бок
  • По умолчанию — по месяцам; при выборе месяца(ев) в фильтре — по неделям, ровно так же, как остальные графики
  • Клик по точке в месячном режиме — переключает в недельный режим именно для этого месяца

Полезно для проверки гипотезы «рост интереса на MPStats опережает рост продаж» — если частота запросов растёт за несколько недель до роста заказов, это видно визуально по сдвигу пиков между двумя линиями.

Топ-50 ключевых запросов MPStats

Источник — та же таблица mpstats_kw_dashboard, но без агрегации по категории: каждая строка — конкретный поисковый запрос (keyword) с его частотой и числом карточек WB, которые под него ранжируются.

  • Показаны все три типа частотности — ВЧ, СЧ и НЧ — каждый в своей колонке, со своим топ-50 и своей шкалой для полосы частоты (иначе шкала ВЧ визуально «съела» бы СЧ/НЧ, у которых частоты на порядок меньше)
  • Частоты суммируются по всем совпавшим неделям/категориям в выбранном периоде и фильтрах, отдельно для каждого типа
  • Внутри каждой колонки сортировка — по убыванию суммарной частоты, показываются первые 50
  • «Поиск по запросу» ищет подстроку в тексте запроса (без учёта регистра) — применяется одинаково ко всем трём колонкам

Тренды по категориям

Автоматический поиск заметных взлётов и падений спроса по категориям, на основе частоты запросов из mpstats_kw_dashboard.

  • Период — только 2025 год. За 2026 год данных всего пара недель, сравнивать на таком коротком отрезке бессмысленно (будет шум)
  • Гранулярность — месяц к месяцу (не неделя к неделе). Недельные данные суммируются в месячные тоталы, дальше сравнивается каждый месяц с предыдущим
  • Какие категории проверяются — если выбраны конкретные категории в фильтре, берутся именно они; если фильтр пуст — топ-12 категорий по суммарному объёму запросов за 2025 год

Формула изменения между двумя месяцами:

change = (текущий_месяц − прошлый_месяц) / прошлый_месяц
  • Рост — фиксируется при change > +40%
  • Падение — фиксируется при change < −30%
  • Пороги разные не просто так: падение ограничено снизу (минимум −100%, спрос не может упасть больше чем до нуля), а рост в принципе не ограничен сверху — поэтому порог падения ниже по модулю, иначе просадки ловились бы значительно реже, чем скачки вверх

Защита от шума:

  • MIN_ABS = 2000 — в обоих сравниваемых месяцах частота запросов должна быть не меньше 2000, иначе скачок игнорируется (низкочастотная категория может дать +200% на паре случайных запросов)
  • Если между месяцами есть разрыв в данных (пропущен календарный месяц) — сравнение не производится через дыру
  • Категория должна иметь минимум 3 месяца данных, иначе не проверяется вообще
  • На категорию берётся только самый сильный рост и самое сильное падение (не первое попавшееся)

Итоговый список сортируется по силе изменения (по модулю) и показывает топ-15 самых заметных событий — самые резкие скачки идут первыми.

Пороги (+40% / −30% / MIN_ABS 2000) — эвристика, подобранная на глаз, не статистический вывод из фактического распределения данных. При желании их можно откалибровать точнее — посчитать реальное распределение изменений месяц-к-месяцу по категориям за 2025 год и выбрать пороги, например, на уровне 90-го/10-го перцентиля.

Как работают фильтры между собой

  • Период (с/по) — применяется ко всем графикам и таблице одинаково
  • Категории (мультивыбор) — точное совпадение для графика заказов (там реальная категория WB), приблизительное совпадение по названию для MPStats-данных (там своя, отдельная таксономия категорий). Явный выбор всегда в приоритете
  • Поиск по запросу — влияет на «Топ-50» и два графика частоты напрямую (текст запроса ищется в самом ключевике); на график заказов и «Тренды по категориям» — косвенно: слова из запроса сверяются с известными названиями категорий WB, и если совпадение находится, заказы и тренды сужаются именно до этой категории (подробнее ниже)
  • Только наши артикулы — влияет только на график «Заказы по неделям» (добавляет линии, не фильтрует); MPStats-данные не привязаны к конкретным SKU, поэтому там этот фильтр не может ничего изменить
  • Клик по строке в «Топ-50» — подставляет текст этого запроса в «Поиск по запросу» и сразу перефильтровывает весь дашборд (та же логика, что и обычный поиск)
  • Клик по категории в «Тренды по категориям» — выбирает эту категорию в фильтре «Категории» и перефильтровывает весь дашборд (та же логика, что и клик по строке в «Топ-50», только выбирает категорию, а не текст поиска)

Если категория выбрана вручную — она в приоритете над поиском для графика заказов и трендов. Если категория не выбрана, но есть текст в поиске — используется категория, «угаданная» по словам запроса.

Текущее состояние всех фильтров сохраняется прямо в адресной строке — можно скопировать ссылку и отправить кому-то, она откроет ровно этот же срез данных.

Если данные не обновлялись дольше ~9 дней (обновление — раз в неделю, порог с запасом) — наверху страницы появится предупреждение, что стоит проверить, отработал ли Дашборд_WB_обновление_данных.py по расписанию.

Как поиск определяет категорию для заказов и трендов

У заказов и MPStats-данных нет текста запроса как такового — есть только название категории. Поэтому при поиске (например, «женские костюмы») слова запроса сверяются с известным списком категорий, и для каждой категории считается, сколько слов запроса в ней встречается. Побеждает категория с максимальным числом совпавших слов.

Это важно, потому что при простом сопоставлении «по любому слову» запрос «женские костюмы» находил бы сразу обе категории — и «Костюмы женские», и «Костюмы мужские» (обе содержат слово «костюмы»). Подсчёт числа совпадений решает эту проблему: «Костюмы женские» содержит оба слова запроса (счёт 2), «Костюмы мужские» — только одно (счёт 1), поэтому выбирается только первая.

Точный пол (женские/мужские) подхватится автоматически, только если сама категория в базе WB уже разделена по полу в названии (например, «Костюмы женские» отдельно от «Костюмы мужские»). Если WB хранит это как одну общую категорию «Костюмы» — фильтр сработает на уровне категории целиком, без разбивки по полу; для точного пола потребуется отдельно доставать эту характеристику из WB_Product_Cards.characteristics.
