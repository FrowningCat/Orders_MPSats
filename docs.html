Как считаются графики

Логика и источники данных для каждого блока дашборда

Заказы по неделям — год к году

Источник — таблица wb_orders_dashboard, заполняется через CSV-отчёт воронки продаж WB (эндпоинт nm-report/downloads). Заказы группируются по ISO-неделям и сравниваются 2025 год к 2026-му на одной шкале недель.

  • Категория (подкатегория) берётся из WB_Product_Cards.subjectName по nm_id
  • Неделя считается по ISO-стандарту прямо из даты заказа — не по полю week_num в таблице, чтобы избежать рассинхрона на границе декабря/января
  • Точка на графике = сумма заказов (шт) за всю неделю по выбранным подкатегориям

Только наши артикулы — не фильтрует график, а добавляет две дополнительные линии зелёным цветом (сплошная — 2025, пунктир — 2026), показывающие только заказы по вашим SKU, поверх общего объёма.

  • «Наш» артикул определяется через цепочку: wb_orders_dashboard.nm_id → WB_Product_Cards.vendorCode → сверка с WB_Working_Articles.article
  • Джойн напрямую по vendor_code из wb_orders_dashboard не используется — это поле там пустое

Динамика суммарной частоты запросов — год к году

Источник — таблица mpstats_kw_dashboard (частотность поисковых запросов WB по неделям), агрегированная по категории (item_name). Сравнение идёт так же, как и с заказами — по ISO-неделям, 2025 к 2026.

  • Без активного поиска — суммарная частота всех запросов категории (total_freq из kw_weekly)
  • При активном поиске «Поиск по запросу» — график переключается на данные из «Топ-50» (конкретные совпавшие запросы), а не всю категорию, чтобы цифры на графике совпадали с тем, что видно в таблице снизу

Динамика частоты запросов MPStats

Тот же источник (mpstats_kw_dashboard), но с разбивкой по типу частотности:

  • ВЧ — высокочастотные запросы
  • СЧ — среднечастотные
  • НЧ — низкочастотные

При активном поиске по запросу график показывает только одну линию — «Совпадения по запросу». Это ограничение источника данных: таблица «Топ-50» (и данные для поиска) изначально собираются только из ВЧ-запросов, поэтому разбивку СЧ/НЧ для конкретного текста запроса показать нельзя — таких данных просто нет на уровне отдельного запроса, только на уровне категории целиком.

Топ-50 ключевых запросов MPStats

Источник — та же таблица mpstats_kw_dashboard, но без агрегации по категории: каждая строка — конкретный поисковый запрос (keyword) с его частотой и числом карточек WB, которые под него ранжируются.

  • Отбираются только запросы с типом ВЧ% (высокочастотные) — остальные типы в эту таблицу не попадают
  • Частоты суммируются по всем совпавшим неделям/категориям в выбранном периоде и фильтрах
  • Сортировка — по убыванию суммарной частоты, показываются первые 50
  • «Поиск по запросу» ищет подстроку в тексте запроса (без учёта регистра)

Тренды по категориям

Автоматический поиск заметных взлётов и падений спроса по категориям, на основе частоты запросов из mpstats_kw_dashboard.

  • Период — только 2025 год. За 2026 год данных всего пара недель, сравнивать на таком коротком отрезке бессмысленно (будет шум)
  • Гранулярность — месяц к месяцу (не неделя к неделе). Недельные данные суммируются в месячные тоталы, дальше сравнивается каждый месяц с предыдущим
  • Какие категории проверяются — если выбраны конкретные подкатегории в фильтре, берутся именно они; если фильтр пуст — топ-12 категорий по суммарному объёму запросов за 2025 год

Формула изменения между двумя месяцами:

change = (текущий_месяц − прошлый_месяц) / прошлый_месяц
  • Рост — фиксируется при change > +40%
  • Падение — фиксируется при change < −30%
  • Пороги разные не просто так: падение ограничено снизу (минимум −100%, спрос не может упасть больше чем до нуля), а рост в принципе не ограничен сверху — поэтому порог падения ниже по модулю, иначе просадки ловились бы значительно реже, чем скачки вверх

Защита от шума:

  • MIN_ABS = 2000 — в обоих сравниваемых месяцах частота запросов должна быть не меньше 2000, иначе скачок игнорируется (низкочастотная категория может дать +200% на паре случайных запросов)
  • Если между месяцами есть разрыв в данных (пропущен календарный месяц) — сравнение не производится через дыру
  • Категория должна иметь минимум 3 месяца данных, иначе не проверяется вообще
  • На категорию берётся только самый сильный рост и самое сильное падение (не первое попавшееся)

Итоговый список сортируется по силе изменения (по модулю) и показывает топ-15 самых заметных событий — самые резкие скачки идут первыми.

Пороги (+40% / −30% / MIN_ABS 2000) — эвристика, подобранная на глаз, не статистический вывод из фактического распределения данных. При желании их можно откалибровать точнее — посчитать реальное распределение изменений месяц-к-месяцу по категориям за 2025 год и выбрать пороги, например, на уровне 90-го/10-го перцентиля.

Как работают фильтры между собой

  • Период (с/по) — применяется ко всем графикам и таблице одинаково
  • Подкатегории (мультивыбор) — точное совпадение для графика заказов (там реальная категория WB), приблизительное совпадение по названию для MPStats-данных (там своя, отдельная таксономия категорий)
  • Поиск по запросу — влияет только на «Топ-50» и два графика частоты (см. описание выше); на график заказов и тренды по категориям не влияет
  • Только наши артикулы — влияет только на график «Заказы по неделям» (добавляет линии, не фильтрует); MPStats-данные не привязаны к конкретным SKU, поэтому там этот фильтр не может ничего изменить
